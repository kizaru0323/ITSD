System Overview

The ITSD Tracker is a robust management information system designed to track, manage, and archive official communications for an IT department. It utilizes a modern 3-tier architecture with a React frontend, an Express/Node.js backend, and a PostgreSQL database.

Core Objectives:

  1. Automation of Record Keeping: Digitizing the tracking of incoming and outgoing documents.
  2. Role-Based Accountability: Ensuring only authorized personnel can delegate and manage specific tasks.
  3. Process Transparency: Providing real-time status updates and activity logs for all system interactions.

Detailed Findings

1. Security Analysis 🛡️

  • CRITICAL VULNERABILITY: Hardcoded Temporary Passwords
    • Problem: In server/index.js (line 941), the system resets user passwords to a hardcoded string "temp123" upon approving a password reset request.
    • Impact: Extremely high. Anyone who knows or guesses that a reset request was approved can log into that user's account using the same universal password. This is a "backdoor" by design.
    • Recommendation: Generate a cryptographically secure random string for each reset and ideally send it via a secure channel (like email) or display it only once to the admin.
  • Vulnerability: Unrestricted File Uploads
    • Problem: The multer configuration in index.js lacks file type validation and size limits.
    • Impact: A malicious user could upload executable scripts (.exe, .php, .js) to the uploads/ directory, potentially leading to Remote Code Execution (RCE).
    • Recommendation: Implement fileFilter in multer to allow only specific safe extensions and set a fileSize limit.
  • Vulnerability: Mixed Authorization Logic
    • Problem: Mixing hardcoded role checks (role === 'admin') and RBAC permission checks.
    • Impact: Creates "shadow permissions" where a user might have the right UI permission but is still blocked by a hardcoded backend role check (or vice versa).
    • Recommendation: Fully migrate to a Permission-Based Access Control (PBAC) model.

2. Functional & Logical Errors ⚙️

  • Data Integrity: Generic Attachment Field
    • Problem: The Communication model uses a TEXT field for JSON data (attachments).
    • Impact: Sequelize/Postgres doesn't validate the structure inside a TEXT field. Corrupted JSON will crash the frontend during parsing.
    • Recommendation: Use the native PostgreSQL JSONB data type for the attachments field.
  • Logic: Volatile Active Session Map
    • Problem: activeSessions are stored in a JavaScript Map.
    • Impact: A server restart logs everyone out visually (shows 0 online) and clears the list of online users.
    • Recommendation: Use Redis for session persistence to survive server restarts.

3. UI/UX Problems 🎨

  • Data Redundancy: Department vs Group
    • Problem: Users have both a groupId (linked to Section) and a department (literal string).
    • Impact: Confusion and data inconsistency. "ITSD Division" in the group table might be "ITSD" in the literal string.
    • Recommendation: Remove the literal department field and always derive the department name from the associated Group.

4. Performance Bottlenecks 🚀

  • Joins and Sorting at Scale
    • Problem: The main communications list performs 4+ joins and returns all records without server-side pagination.
    • Impact: Once the system hits 10,000+ records, page loads will stall.
    • Recommendation: Implement server-side pagination with limit and offset in Sequelize queries.

Complete System Flow (Student's Guide)

Think of the ITSD Tracker like a Postal Office with a Security Guard.

  1. Entry (Login): You come to the door and show your ID card (Email/Password). The Guard (Auth Middleware) checks if you are "Active". If yes, he gives you a Special Keychain (JWT) that lists your name and what rooms you can enter (Permissions).
  2. Dropping a Package (New Communication):
    • You fill out a form.
    • You pick a Team Leader (Section Head) responsible for the package.
    • The system gives it a Tracking Number.
  3. Sorting Room (Admin Review):
    • Admins and Section Heads see the package in their "Pending" list.
    • The Section Head assigns specific workers (Personnel) to handle it.
  4. Action (Processing):
    • Workers mark it as "COMPLETED" when done.
  5. History (Archive):
    • The package moves to the "Processed" cabinet for history.

Recommendations

  1. URGENT: Replace hardcoded "temp123" password with a random generator.
  2. Security: Implement multer file filters and size limits.
  3. Architecture: Switch to Server-Side Pagination for communication lists.
  4. Data: Use JSONB for attachments and remove redundant string fields.
